# 모임코드 분실 동선 — 분석 보고서 (2026-05-20, v7 club 도입 반영)

## 1. 현재 상태 (As-is)

| 역할 | 인증 키 | 분실 시 동선 |
|---|---|---|
| 회원 | 휴대폰 뒤 4자리 (식별자, 인증 아님) | 모임 목록에서 **공개(is_open=true) 모임만** 진입 가능. 비공개 모임의 코드를 잊으면 진입 불가. |
| 모임장 | **모임(club) 코드 + 비밀번호** (둘 다 평문, club 테이블 — v7 부터 인증이 club 단위로 통일) | **자가 복구 동선 없음.** 코드/비번 둘 다 잊으면 본인이 만든 모임도 못 만짐. |

> ⚠️ v7 이전엔 인증 키가 시즌 단위 (meeting.code/pw) 였음. v7 부터 club 단위로 옮겨졌고, 같은 코드 1개로 그 모임의 모든 시즌을 관리. 따라서 분실 영향이 **시즌 1개 → 모임 전체**로 더 커짐.

발급 직후 코드는 (a) toast 한 줄 + (b) 결과 박스에 표시. 회원에게는 모임장이 직접 전달.

## 2. 문제 진단

1. **회원 분실** — 비공개 시즌만 해당. 공개 시즌은 목록에서 찾으면 되니 큰 문제 아님. 단 본인이 이미 한 번 참여한 비공개 시즌(member_book 행 있음)도 코드 잊으면 재진입 불가.
2. **모임장 분실** — 치명적. 운영자가 Supabase 콘솔에서 meeting 테이블을 직접 조회해서 알려주는 핫라인 외엔 답이 없음. 4자리/이메일/SMS 같은 신원 묶는 키가 없어서 "본인 인증 → 코드 회수" 회로 자체가 없음.
3. **발급 직후 노출 강도 약함** — 결과 박스에서 다른 페이지로 이동하면 사라짐. 모임장이 메모하지 않으면 끝.

## 3. 후보 해결책

| 안 | 비용 | 효과 | 영향 범위 |
|---|---|---|---|
| **A. 발급 결과 강조 + 영속 메모** | 낮음 (UI만) | 발급 직후 분실 방지 (사후 복구 X) | 프런트만 |
| **B. 회원 자동 매핑** — phone4 입장 시 본인이 참여한 적 있는 시즌은 비공개여도 목록에 노출 | 중간 | 회원의 비공개 시즌 재진입 가능 | 프런트 + RT.loadMeetings 로직 |
| **C. 모임장 복구 4자리** — 시즌 발급 시 모임장 휴대폰 4자리(또는 별도 단어) 등록 → 그 4자리만 입력하면 본인 시즌 목록 + 코드/비번 노출 | 중간 | 모임장 분실 시 자가 복구 가능 | meeting 컬럼 추가 + 프런트 + 시즌 발급 폼 |
| **D. 운영자 콘솔** — 별도 페이지에 마스터 비번으로 전체 meeting 조회 | 낮음 (단 보안 위험) | 운영자 부담 경감 | 프런트(노출만) — anon 키로 전체 노출은 비추 |
| **E. 코드 즉시 노출 인터페이스** — "내 모임 보기" 단축키, 클립보드 복사, "메모장으로 저장" 다운로드 | 낮음 | A 와 같은 사전 예방 | 프런트만 |

## 4. 권장안 (2단계)

### 4-1. 단기(이번 PR) — 사전 예방만 강화
- 발급 결과 박스에 **⚠️ 분실 시 복구 불가 경고** + **클립보드 복사 버튼** 추가 (이번 PR 에 기본 경고는 이미 추가됨).
- 발급 결과를 localStorage `rt:myseasons` 에 누적 저장하고, 첫 화면 모임장 영역에 "내가 발급한 시즌(이 브라우저)" 목록 표시.

### 4-2. 중기(별도 PR) — 자가 복구
- **회원 자동 매핑(B)**: ONLINE 모드 RT.loadMeetings 에 `phone4` 파라미터 추가 → meeting + (member_book where phone4=me) UNION 으로 비공개 + 본인 참여 시즌을 함께 반환.
- **모임장 복구 4자리(C)**:
  - 마이그레이션: `meeting.host_phone4 text` 컬럼 추가 (nullable).
  - 시즌 생성 폼에 "복구용 휴대폰 4자리" 필드 추가(기억 쉬운 4자리).
  - 시즌 관리 게이트 옆에 "코드 잊었어요" 링크 → 4자리 입력 → matching meeting 들의 (시즌명·코드·비번) 표시.
  - 보안: 어차피 데모 등급 RLS(anon 허용)라 4자리 인증은 약하지만 현 모델 정합.

### 4-3. 장기(필요 시) — 인증 강화
- Supabase Auth(이메일 매직 링크 등)로 모임장 신원 묶기. RLS 좁히기. 이 단계는 사내 소규모 모델 벗어날 때.

## 5. 즉시 분석 결과 — "이걸 반영해서 뭘 분석하면 좋을지"

이 사건이 시사하는 것:
1. **신원 묶는 키가 없는 모델의 한계.** "휴대폰 4자리 = 식별자, 인증 아님" 모델은 사용자 흐름이 적을 때만 작동. 모임장처럼 운영 권한이 있는 역할에는 신원 회수 통로가 반드시 필요.
2. **발급 = 단발 행위.** 발급 결과를 사용자가 받아 적지 않으면 휘발. UI 가 "한 번 보여주고 끝"인 모든 비밀값(코드/비번)은 보존·복구 동선과 페어로 설계돼야 함.
3. **공개/비공개 시즌의 의미.** is_open=false 는 본 PR 기준 "모임 목록 미노출"인데, 동시에 "코드 잊으면 끝"이 됨. is_open=false 의 정책 의도(미공개 시즌의 보안 강도)와 실제 효과(복구 불가)가 일치하는지 재확인 필요.

이번 PR 에는 **(1) 발급 결과 박스 경고 강화** 만 반영. 4-2 자가 복구는 사용자 결정 후 별도 PR 진행 권장.
